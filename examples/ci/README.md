# CI parallelization examples

This is where CI Shard Advisor connects to real cloud execution. Parallelizing
Cypress *is* splitting a run into N containers on separate, separately billed
machines — and keeping one report per container.

```text
   ┌── container 1 ──┐
   ├── container 2 ──┤   each runs its own spec list (billed separately)
   ├──     …     ────┤   → one mochawesome report per container
   └── container N ──┘
              │
   one file per container  ← CI Shard Advisor reads these
```

## The generated workflows (start here)

- [`github-actions-cypress.yml`](github-actions-cypress.yml) — what
  `--format github` emits for the sample run: one job per container, each
  running **exactly its duration-optimal spec list** and uploading its own
  mochawesome JSON. This file was generated by the tool itself.
- [`bitbucket-pipelines-cypress.yml`](bitbucket-pipelines-cypress.yml) — the
  same run through `--format bitbucket`: one parallel step per container,
  keeping its own report as an artifact. Also generated by the tool
  (`ci-shard-advisor samples/cypress-containers/container-*.json --setup 45s
  --format bitbucket`).

A **live** example: this very repo's [CI](../../.github/workflows/ci.yml) runs
its Cypress suite as a 2-container matrix, one mochawesome artifact each —
download them from any run and feed them back to the advisor.

## The loop

1. Run your pipeline → download the per-container report artifacts.
2. Feed them all to the advisor:

   ```bash
   ci-shard-advisor container-*.json --setup 45s --price 0.008
   ```

   …or drop the files into the web app — same analysis, in your browser.
3. It anchors to where you are today and shows your moves: rebalance for free,
   the same wait for less money, or the knee where containers stop paying off —
   each with its `--spec` lists and, via `--format github|bitbucket`, the full
   paste-ready workflow.

## Note on the split

Naive splitting (same number of specs per container) balances by **count**, not
duration. The advisor closes that gap: the emitted spec lists are optimal by
**duration**, at spec-file granularity — a spec never straddles containers, so
its internal order is always preserved.

> The `github-actions.yml` / `github-actions-per-shard.yml` /
> `bitbucket-pipelines.yml` templates remain from the parked Playwright reader.
